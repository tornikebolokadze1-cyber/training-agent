# AI აგენტების განვითარება — ჯგუფური ქოუჩინგ-სესია

> ჯგუფი #1 -- ლექცია #10
> თარიღი: 2026-04-14

## შეჯამება

აქ მოცემულია ექსპერტის ანალიზის სრული, დეტალური და ზუსტი თარგმანი ქართულ ენაზე, ორიგინალის სტრუქტურისა და სიღრმის სრული დაცვით:

## ლექციის შეჯამება: AI აგენტების განვითარება — ჯგუფური ქოუჩინგ-სესია

### 1. განხილული მთავარი თემები

ეს სესია წარიმართა არა ტრადიციული ლექციის, არამედ **ჯგუფური ქოუჩინგის/მენტორინგის ფორმატში**. ინსტრუქტორი (თორნიკე) ინდივიდუალურად მუშაობდა მონაწილეებთან — კობასთან, ლაშასთან, ქეთისთან, ნინოსთან, შორენასთან და მაკასთან — მათ მიმდინარე AI აგენტების პროექტებზე. მთავარი თემები მოიცავდა: მულტი-აგენტურ არქიტექტურას, LLM მოდელების შერჩევას, აგენტების მეხსიერების/კონტექსტის სისტემებს, API მენეჯმენტს, კონტენტის ავტომატიზაციას და პროექტის უსაფრთხოებას/გაშვებას (deployment).

### 2. ახსნილი ძირითადი კონცეფციები და იდეები

**A. მულტი-აგენტური არქიტექტურა (განხილული კობასთან)**
*   განიხილეს პროექტის ფაზებად დაყოფის და **თითოეული ფაზისთვის ცალკე აგენტის** მინიჭების იდეა.
*   ინსტრუქტორმა წარადგინა **"მეთაური აგენტის" (commander agent)** კონცეფცია (მოხსენიებული როგორც "paperclip"-სტილის სისტემა) — ერთი მთავარი ორკესტრატორი აგენტის, რომელიც დინამიურად ქმნის და მართავს ქვე-აგენტებს დავალების საჭიროებიდან გამომდინარე. ეს ბევრად უფრო მოწინავე მიდგომაა, ვიდრე ყველა აგენტის წინასწარ, ხელით განსაზღვრა.
*   რეკომენდაცია: კობამ ჯერ უნდა მიიღოს გამოცდილება ერთი აგენტის გამოყენებით, რათა მან თავად მოახდინოს მულტი-აგენტური სამუშაო პროცესის (flow) თვითორგანიზება, და მხოლოდ ამის შემდეგ ისწავლოს ფორმალური "მეთაურის" მიდგომა.
*   ახსნილი საკვანძო განსხვავება: LLM (მაგ. DeepSeek, Gemini) შეიძლება ემსახურებოდეს აგენტის "ტვინს" და ის დამოუკიდებელია დეველოპმენტის ხელსაწყოებისგან (მაგ. Claude VS Code-ში). ეს როლები ერთმანეთისგან განცალკევებულია.

**B. LLM მოდელის შერჩევა და საიმედოობა (კობასთან)**
*   DeepSeek განხილულ იქნა როგორც ძლიერი, მაგრამ ზოგჯერ არასაიმედო მოდელი გარკვეული ამოცანებისთვის. ინსტრუქტორმა აღნიშნა, რომ ცუდი შედეგი ხშირად **თავად მოდელის** ბრალია და არა პრომპტის.
*   **OpenRouter** აღიწერა როგორც სასარგებლო შუამავალი (intermediary) დამწყებთათვის/ტე
გარკვეული ამოცანებისთვის. ინსტრუქტორმა აღნიშნა, რომ ცუდი შედეგი ხშირად **თავად მოდელის** ბრალია და არა პრომპტის.
*   **OpenRouter** აღიწერა როგორც სასარგებლო შუამავალი (intermediary) დამწყებთათვის/ტესტირებისთვის, თუმცა ერთი გაფრთხილებით: როგორც შუალედურმა ფენამ, მან შეიძლება ზოგჯერ გამოიწვიოს მონაცემთა შეუსაბამობა ან ცარიელი პასუხები. პროდაქშენისთვის რეკომენდებულია მოდელის API-სთან პირდაპირი წვდომა.
*   პრაქტიკული რჩევა: პრომპტებში ყოველთვის მიუთითეთ **ზუსტი თარიღი**, რათა მოდელმა არ დააბრუნოს მოძველებული ან სეზონურად შეუსაბამო მონაცემები.
*   მოდელის შეცვლის სტრატეგია: როდესაც VS Code-ში დეველოპმენტისას Claude-დან GPT-ზე ან Gemini-ზე გადადიხართ, ახალ მოდელს პირდაპირ უნდა უთხრათ: "ეს პროექტი აშენდა Claude-ის მიერ — გააგრძელე მუშაობა არსებული ფუნქციონალის გაფუჭების გარეშე."

**C. გრაფული ცოდნა (Graph Knowledge) და მოწინავე მეხსიერება/RAG (ლაშასთან)**
*   წარმოდგენილი იქნა კრიტიკულად მნიშვნელოვანი ახალი ტრენდი: **გრაფული ცოდნა (GraphNology/Knowledge Graphs)**, როგორც სტანდარტული RAG-ის (Retrieval-Augmented Generation) განახლება.
*   სტანდარტული RAG (მაგ. Qdrant ვექტორული ძიება) პოულობს დოკუმენტებს, მაგრამ არ აქვს მიმართებების (relational) გაგების უნარი. ინსტრუქტორმა ახსნა, რომ ბოლო კვირებში გაჩნდა **ახალი ტრენდი** — პროექტის მთელი ცოდნის შენახვა **ვიკი-სტილის ურთიერთდაკავშირებული კვანძების (nodes)** სახით (ვიკიპედიის ბმულების სტრუქტურის მსგავსად), რაც უზრუნველყოფს კონტექსტის ბევრად უფრო ღრმა გაგებას.
*   ეს მიდგომა აგენტებს აძლევს უკეთეს გრძელვადიან მეხსიერებას და პროექტის აღქმის უნარს, რაც გადამწყვეტია რთული, უწყვეტი სისტემებისთვის.
*   ნახსენები რესურსები: სპეციალურად შერჩეული კოლექცია, რომელიც მოიცავს მეხსიერების/RAG ბიბლიოთეკებს და **LLM Maintained Wiki**-ს (სასწავლო რესურსების რეპოზიტორი).
*   ლაშას ამჟამინდელი სტეკი: PostgreSQL, Redis, Qdrant — სტანდარტულია, მაგრამ არასაკმარისი მისი ტრეიდინგ-სისტემის სიღრმისთვის.

**D. კონტენტის ავტომატიზაციის აგენტის არქიტექტურა (ლაშასთან)**
*   ლაშამ შექმნა სოციალური მედიის ავტომატიზაციის მომუშავე აგენტი, რომელიც: ეძებს ყოველ
არასაკმარისი მისი ტრეიდინგ-სისტემის სიღრმისთვის.

**D. კონტენტის ავტომატიზაციის აგენტის არქიტექტურა (ლაშასთან)**
*   ლაშამ შექმნა სოციალური მედიის ავტომატიზაციის მომუშავე აგენტი, რომელიც: ეძებს ყოველდღიურ ნიშურ ინფორმაციას, აგენერირებს სურათებს (Gemini-ს გამოყენებით) და ავტომატურად აქვეყნებს პოსტებს LinkedIn-ზე, Facebook-სა და Instagram-ზე — გამოქვეყნებამდე Telegram-ში წინასწარი ნახვის (preview) ფუნქციით.
*   ინსტრუქტორმა ურჩია ამ სისტემის მნიშვნელოვნად გაფართოება **მრავალფორმატიან კონტენტის ფაიფლაინად (pipeline)**:
    1. ფოტო/სურათის კონტენტი
    2. ინფოგრაფიკული კონტენტი (**NotebookLM.py** Python ბიბლიოთეკის გამოყენებით)
    3. ვიდეო პრეზენტაციის კონტენტი (NotebookLM-ის გამოყენებით — 5-წუთიანი AI-გახმოვანებული პრეზენტაციები სლაიდებით)
    4. სტანდარტული ვიდეო კონტენტი (Gemini)
    5. ავატარის ვიდეო კონტენტი (ვირტუალური პერსონაჟი, რომელიც კითხულობს ტექსტს)
    6. შერეული/სხვა ფორმატები
*   რეკომენდებულია კონტენტის ტიპების პროპორციულად გადანაწილება (თითოეული დაახლ. 15-20%), რათა მიღწეულ იქნას მაქსიმალური მრავალფეროვნება და ჩართულობა (engagement).

**E. ხედვა/სურათის ანალიზი პროდუქტის დიფერენციაციისთვის (ქეთისთან)**
*   ქეთი ქმნის აგენტს ჩანთების/აქსესუარების ბიზნესისთვის და სჭირდება, რომ აგენტმა შეძლოს პროდუქტების გარჩევა ფერისა და სტილის მიხედვით.
*   ინსტრუქტორმა ურჩია **Gemini-ს ხედვის (vision) შესაძლებლობების** გამოყენება და მისი პირდაპირ **Google AI Studio**-ში დატესტვა, სანამ მას აგენტის ფაიფლაინში დაამატებს.
*   Google AI Studio ნაჩვენები იქნა როგორც სრულყოფილი სატესტო გარემო, რომელიც მოიცავს: ჩატს, სურათების გენერაციას, ვიდეოს გენერაციას, მუსიკის გენერაციას, რეალურ დროში ხმას, აპლიკაციის პროტოტიპირებას ("Build" ფუნქცია) და აუდიოს ტრანსკრიფციას.
*   Google AI Studio-ს "Build" ფუნქცია იძლევა მინი-აპლიკაციების პროტოტიპირების საშუალებას კონკრეტული Google AI ფუნქციების არჩევით.
*   Pinterest API განიხილეს როგორც სურათების გენერაციისთვის საჭირო რეფერენს-სურათების მოძიების ეფექტური წყარო.
*   ნახსენები იყო **უნარების (skills) სისტემის** კონცეფცია — აგენტისთვის სტ
ების არჩევით.
*   Pinterest API განიხილეს როგორც სურათების გენერაციისთვის საჭირო რეფერენს-სურათების მოძიების ეფექტური წყარო.
*   ნახსენები იყო **უნარების (skills) სისტემის** კონცეფცია — აგენტისთვის სტრუქტურირებული შესაძლებლობების დამატება დავალებების უკეთესი მარშრუტიზაციისთვის (routing).

**F. ვიდეოს ტრანსკრიფცია ქართული ენის მხარდაჭერით (ნინოსთან)**
*   ნინომ სცადა ვიდეო ტრანსკრიფციის აგენტის შექმნა Whisper-ის გამოყენებით (რაც წარუმატებელი აღმოჩნდა ქართული ენის ცუდი მხარდაჭერის გამო) და Google AI Studio API-ით.
*   ინსტრუქტორმა დაადასტურა, რომ **Gemini შესანიშნავად უმკლავდება ქართული ენის ტრანსკრიფციას**, მათ შორის 3-საათიანი ლექციების ჩანაწერების შემთხვევაშიც.
*   ახსნილი იქნა უსაფრთხოების კრიტიკული გაკვეთილი: **არასოდეს ჩასვათ API გასაღებები პირდაპირ ჩატის ფანჯარაში**. ყოველთვის შეინახეთ ისინი პროექტის `.env` ფაილში (გარემოს ცვლადები - environmental variables) და გამოიძახეთ იქიდან. ნაბიჯ-ნაბიჯ აიხსნა პროცესი: API გასაღების მიღება → გასაღების შექმნა Google AI Studio-ში → `.env` ფაილის გახსნა VS Code-ის ფაილების ექსპლორერში → გასაღების ცვლადის იქ დამატება.
*   ინსტრუქტორმა ნინოს შესთავაზა დახმარება API წვდომისთვის Google Cloud-ის $300-იანი კრედიტის გააქტიურებაში.

- GitHub-ზე ბექაფი სავალდებულოა ნებისმიერი პროექტის deployment-მდე — პროექტის მხოლოდ ლოკალურად შენახვა სერიოზული რისკია.

- OpenRouter კარგია სასწავლებლად, მაგრამ პროდაქშენის საიმედოობისთვის გადადით პირდაპირ API წვდომაზე.
- ღრმა აუდიტის პრომპტები (და არა ზოგადი) აუცილებელია deployment-მდე — სტანდარტული AI თვით-აუდიტი კრიტიკულ პრობლემებს ვერ ამჩნევს.

- [[Content Automation Pipeline]] -- სოციალური მედიის ავტომატიზაციის სისტემა, რომელიც ყოველდღიურ ნიშურ ინფორმაციას ეძებს, სურათებს აგენერირებს Gemini-ს მეშვეობით და ავტომატურად აქვეყნებს LinkedIn-ზე, Facebook-სა და Instagram-ზე — Telegram-ში preview-ს შემდეგ.
- [[Gemini]] -- ქართულენოვანი ვიდეო ლექციების ტრანსკრიფცია — Whisper-ის ჩანაცვლება, რომელსაც ქართული ენის მხარდაჭერა სუსტი ჰქონდა.
- [[Deep Audit Prompt]] -- პროექტის deployment-მდე სრული შემოწმება: ENV საიდუმლოებები, rate limiting, ავტომატიზებული ტესტები, სარეზერვო სისტემა, ინსტალერი, UI და იურიდიული დოკუმენტაცია — შორენას პროექტმა 8/10 მიიღო, ქულები დააკლდა ამ ხარვეზების გამო.
- [[Claude]] -- N8N workflow-ების ავტომატური გენერაცია VS Code-ში Codex-ის მეშვეობით — ექსპერტებიც კი ამ მიდგომას ხელით კეთებას ამჯობინებენ.
- [[NotebookLM]] -- კონტენტის ავტომატიზაციის ფაიფლაინში ინფოგრაფიკებისა და 5-წუთიანი AI-გახმოვანებული ვიდეო პრეზენტაციების სლაიდებით გენერაცია.
- [[Google AI Studio]] -- Gemini-ს Build ფუნქციის გამოყენება მინი-აპლიკაციების პროტოტიპირებისთვის — კონკრეტული ფუნქციების არჩევა (სურათის ანალიზი, ტრანსკრიფცია, TTS) სწრაფი ვალიდაციისთვის.
- [[Knowledge Graph]] -- ტრეიდინგ-სისტემის AI აგენტისთვის გრძელვადიანი მეხსიერების სისტემის მშენებლობა ვიკი-სტილის ურთიერთდაკავშირებული კვანძებით, PostgreSQL + Redis + Qdrant სტეკის გასაღრმავებლად.
