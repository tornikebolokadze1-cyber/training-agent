## Gap Analysis

აქ მოცემულია ექსპერტის ანალიზის სრული და დეტალური თარგმანი ქართულ ენაზე, ორიგინალის სტრუქტურის, სიღრმისა და პროფესიონალური ტონის სრული დაცვით:

**ექსპერტის ანალიზი:**

### 1. სწავლების ხარისხი — ახსნის სიცხადე და გასაგებობა

**ძლიერი მხარეები:**
*   **ანალოგიების შესანიშნავი გამოყენება არატექნიკური აუდიტორიისთვის** (Photoshop-ი IDE-სთვის, USB კაბელი MCP-სთვის, ყალიბი/შაბლონი პროექტის ინიციალიზაციისთვის, უბნის მაღაზია ჰიპერმარკეტის წინააღმდეგ Claude desktop-ისა და VS Code-ის შესადარებლად).
*   **გულწრფელი ენთუზიაზმი**, რომელიც გადამდებია.
*   **კარგი ინსტინქტი** მონაწილეთა რეალური პროექტების განსახილველად (ტარიელის სასტუმროს მონიტორინგი).
*   **გულწრფელობა სირთულეებთან დაკავშირებით** ("რომი ერთ დღეში არ აშენებულა").
*   **კარგი აღქმა** იმისა, რომ ჯგუფში მონაწილეებს განსხვავებული დონის უნარები აქვთ.

**სუსტი მხარეები:**
*   **მკვეთრად არაორგანიზებული მსვლელობა:** ლექცია თემიდან თემაზე ხტება მკაფიო გადასვლების გარეშე. 45 წუთის განმავლობაში ინსტრუქტორი შეეხო Claude-ის ინტერფეისებს, VS Code-ის დაყენებას, MCP სერვერებს, სკრეპინგის (scraping) მეთოდებს, n8n-ს, ფასებს, დაგეგმვას (scheduling), Skills-ს, Antigravity-სა და პროექტის ინიციალიზაციას — ისე, რომ არცერთი კონკრეტული თემის ახსნა არ დაუსრულებია.
*   **მუდმივი თვით-შეწყვეტა:** ფრაზები, როგორიცაა "მაგრამ მანამდე, მოდით გეტყვით...", ათობით ჯერ გვხვდება, რაც ქმნის დაუსრულებელი ახსნების ჩაშენებულ (nested) სტრუქტურას, რომლის გაყოლაც შეუძლებელია.
*   **ბუნდოვანი ტექნიკური ახსნები:** MCP ახსნილი იყო მეტაფორულად, მაგრამ არა საკმარისად კონკრეტულად იმისთვის, რომ მონაწილეებმა რეალურად შეძლონ მისი დაყენება. USB-ის ანალოგია ადვილად დასამახსოვრებელია, მაგრამ არ გადმოსცემს, თუ რას აკეთებს MCP ტექნიკურად.
*   **დამაბნეველი ტერმინოლოგია:** ინსტრუქტორი გამუდმებით ურევს ერთმანეთში "Claude Code"-ს (პროდუქტს) და "Code"-ს (ჩანართს დესკტოპ აპლიკაციაში), და ერთ მომენტში, როგორც ჩანს, Claude Code-ს აიგივებს თავად Claude-ის დესკტოპ აპლიკაციასთან.
*   **არანაირი ვიზუალური მასალა ან სლაიდები:** მთელი ლექცია წარმოადგენს ლაივ-რეჟიმში ინსტრუმენტების ე
იაში), და ერთ მომენტში, როგორც ჩანს, Claude Code-ს აიგივებს თავად Claude-ის დესკტოპ აპლიკაციასთან.
*   **არანაირი ვიზუალური მასალა ან სლაიდები:** მთელი ლექცია წარმოადგენს ლაივ-რეჟიმში ინსტრუმენტების ეკრანის გაზიარებას (screen-sharing) სიტყვიერი ახსნით. არ არის წინასწარ მომზადებული დიაგრამები, კონცეპტუალური რუკები ან საცნობარო არქიტექტურა.
*   **გამეორებადი კონტენტი:** Antigravity-სა და Skills-ის სექციები ტრანსკრიპტში სიტყვასიტყვით განმეორდა (მე-3 და მე-4 ნაწილები შეიცავს დუბლირებულ კონტენტს), რაც მიუთითებს დროის ცუდ მართვაზე ან ჩაწერის პრობლემებზე.

### 2. კრიტიკული ხარვეზები — გამოტოვებული ან არასაკმარისად განხილული მნიშვნელოვანი თემები

**მთავარი გამოტოვებული საკითხები:**
*   **უსაფრთხოების/კონფიდენციალურობის განხილვის არარსებობა:** ინსტრუქტორმა აჩვენა რეალური API გასაღებები, Gmail-ზე წვდომა და პერსონალური მონაცემების სკრეპინგი მონაცემთა კონფიდენციალურობის, GDPR-ის გავლენის ან უსაფრთხოების საუკეთესო პრაქტიკის ნებისმიერი განხილვის გარეშე. ქართველი ბიზნეს მენეჯერებისთვის, რომლებიც მომხმარებელთა მონაცემებს ამუშავებენ, ეს კრიტიკული ხარვეზია.
*   **ხარჯების ანალიზის არარსებობა:** ფასები ნახსენები იყო ($20/$100/$200), მაგრამ რეალისტური ხარჯების პროგნოზი არ გაკეთებულა. რა ჯდება ტიპური აგენტის პროექტის აწყობა? როგორია მიმდინარე საოპერაციო ხარჯები? ბიზნეს გადაწყვეტილების მიმღებთათვის ეს აუცილებელია.
*   **არ არის ახსნილი დიფლოიმენთი (Deployment):** ინსტრუქტორმა ახსენა, რომ "უნდა ატვირთოთ სერვერზე", მაგრამ არასდროს აუხსნია რას ნიშნავს ეს, რა სერვისები უნდა გამოიყენონ (Railway, Vercel, AWS და ა.შ.) ან რა ჯდება ეს.
*   **შეცდომების მართვის/დიბაგინგის (debugging) საფუძვლების არარსებობა:** როდესაც რაღაც არასწორად მიდის (და ეს აუცილებლად მოხდება), მონაწილეებს არ აქვთ არანაირი ჩარჩო შეცდომების გასაგებად ან კომუნიკაციისთვის.
*   **ეთიკური/სამართლებრივი ჩარჩოს არარსებობა:** კონკურენტების მონაცემების სკრეპინგი, სოციალური მედიის მონიტორინგის ავტომატიზაცია, Playwright-ის გამოყენება პლატფორმის შეზღუდვების გვერდის ავლის მიზნით — ამ ყველაფერს აქვს სამართლებრივი შედეგები, რაც სრულიად იგნო
ის მონაცემების სკრეპინგი, სოციალური მედიის მონიტორინგის ავტომატიზაცია, Playwright-ის გამოყენება პლატფორმის შეზღუდვების გვერდის ავლის მიზნით — ამ ყველაფერს აქვს სამართლებრივი შედეგები, რაც სრულიად იგნორირებული იყო.
*   **AI მოდელების შედარების არარსებობა:** Claude წარმოდგენილი იყო როგორც უდავოდ საუკეთესო, კონკრეტული გამოყენების შემთხვევებისთვის GPT-4-თან, Gemini-სთან ან სხვებთან ნიუანსური შედარების გარეშე.
*   **პრაქტიკული სავარჯიშოების არარსებობა:** მონაწილეები უყურებდნენ, თუ როგორ აკეთებდა ინსტრუქტორი ყველაფერს. ლექციის განმავლობაში არავის არაფერი აუწყვია/შექმნილა პრაქტიკულად.

### 3. ტექნიკური სიზუსტე

**გამოვლენილი პრობლემები:**
*   **მტკიცება: "Claude Code უდავოდ ყველაზე ძლიერია"** — ეს საკამათოა. გარკვეული ამოცანებისთვის, OpenAI-ს Codex/GPT-4 ან თუნდაც Cursor შეიძლება სჯობდეს Claude Code-ს. ასეთი აბსოლუტური ფორმულირება შეცდომაში შემყვანია.
*   **მტკიცება: "Claude Code-ისა და n8n-ის გარდა ყველაფერი მაიმუნობაა"** — ეს უგულებელყოფს ისეთ ლეგიტიმურ ინსტრუმენტებს, როგორიცაა Cursor, Windsurf, Replit Agent, Lovable და სხვა, რომლებიც შესაძლოა უფრო შესაფერისი იყოს გარკვეული მონაწილეებისთვის.
*   **მტკიცება: "Gemini ვერ გაუწევს კონკურენციას Claude-ს ან ChatGPT-ს კოდირებასა და ღრმა აზროვნებაში"** — Gemini 2.5 Pro-მ აჩვენა კონკურენტუნარიანი ან უკეთესი შედეგები კოდირების რამდენიმე ბენჩმარკში. ეს ინფორმაცია მოძველებული/არაზუსტია.
*   **MCP-ის ახსნა ზედმეტად გამარტივებულია:** MCP არ არის უბრალოდ "USB კაბელი" — ეს არის კონკრეტული პროტოკოლი კლიენტ-სერვერის არქიტექტურით და ეს ახსნა არ ამზადებს მონაწილეებს იმ შეცდომის შეტყობინებების ან კავშირის პრობლემების გასაგებად, რომლებსაც ისინი აუცილებლად წააწყდებიან.
*   **Claude Desktop App-ის ფუნქციების აღრევა Claude Code-თან:** Claude Code (CLI ინსტრუმენტი) და Claude Desktop App-ის "Code" ჩანართი დაკავშირებული, მაგრამ განსხვავებული პროდუქტებია. ეს განსხვავება არასდროს განმარტებულა.
*   **"ქართული ტელეფონის ნომრები მუშაობს Claude-ის რეგისტრაციისთვის"** — ეს ითქვა, მაგრამ რამდენიმე მონაწილემ აღნიშნა, რომ არ მუშაობდა და ინსტრუქტორმა აღიარა,
ა. ეს განსხვავება არასდროს განმარტებულა.
*   **"ქართული ტელეფონის ნომრები მუშაობს Claude-ის რეგისტრაციისთვის"** — ეს ითქვა, მაგრამ რამდენიმე მონაწილემ აღნიშნა, რომ არ მუშაობდა და ინსტრუქტორმა აღიარა, რომ არ იცოდა რატომ. ეს ლექციამდე უნდა შემოწმებულიყო.

### 4. პედაგოგიკური რეკომენდაციები

**სტრუქტურა:**
*   **შექმენით მკაფიო დღის წესრიგი** დასაწყისში დანომრილი თემებითა და სავარაუდო დროით.
*   **გამოიყენეთ ჩარჩო: "უთხარი მათ, რის თქმას აპირებ; უთხარი; უთხარი მათ, რა უთხარი".**
*   **დაასრულეთ ერთი თემა მეორის დაწყებამდე** — ჩაშენებული, შეწყვეტილი ახსნები ნომერ პირველი პრობლემაა.
*   **მოამზადეთ სლაიდები/ვიზუალური დიაგრამები:** მარტივი არქიტექტურული დიაგრამა, რომელიც აჩვენებს Claude Desktop ↔ VS Code ↔ MCP ↔ გარე სერვისებს, 20 წუთიანი სიტყვიერი ახსნის ტოლფასი იქნებოდა.

**სავარჯიშოები:**
*   **ლექციას სჭირდება პრაქტიკული კომპონენტი:** 15 წუთიც კი, სადაც ყველა ერთად დააინსტალირებს VS Code-ს და Claude-ის ექსთენშენს (extension), მკვეთრად გააუმჯობესებს მასალის ათვისებას.

## ნაწილი I — სწავლების ხარისხი (პუნქტები 1-6 შეჯამება)

**სიცხადე (პუნქტი 1):** ლექტორის მიერ მოყვანილი ანალოგიები (MCP = USB კაბელი, IDE = Photoshop კოდისთვის) ნამდვილად ეფექტურია და აჩვენებს რეალურ პედაგოგიკურ უნარებს. თუმცა, თხრობის სტილი ჰგავს „ცნობიერების ნაკადს“ (stream-of-consciousness), რაც მსმენელებს უქმნის ინფორმაციის დამოუკიდებლად რეკონსტრუქციის ტვირთს. ახსნის ხარისხი უკუპროპორციულია თემის სირთულისა — მარტივი კონცეფციები ახსნილია ძალიან ნათლად, ხოლო რთული თემები (ტოკენების მართვა, პროექტის დიფლოიმენტი/გაშვება) მხოლოდ ნახსენებია და არა ნასწავლი.

**კრიტიკული ხარვეზები (პუნქტი 2):** ყველაზე საშიში ხარვეზი არის შეცდომების მართვის (error handling), დიბაგინგისა და მარცხის სცენარების სრული უგულებელყოფა. კურსის აგება დაპირებაზე: „თქვენ შეგიძლიათ შექმნათ აგენტები კოდის წერის გარეშე“, იმ ფონზე, რომ ნულოვანი დრო ეთმობა იმას, თუ რა უნდა ქნან, როცა აგენტი შეცდომას უშვებს — ქმნის „მაღალი თავდაჯერებულობის, მაგრამ დაბალი კომპეტენციის“ რისკს. მეორადი კრიტიკული ხარვეზებია: API გასაღებების უსაფრთხოება, დიფლოიმენტი, ვებსკრეპინგის იურიდიული საკითხები და ხარჯების დათვლა.

**ტექნიკური სიზუსტე (პუნქტი 3):** ფრეიმვორქის დონეზე შინაარსი გამართულია. რისკის ზონები დეტალებშია: არატექნიკური მომხმარებლებისთვის MCP-ის კონფიგურაციის სირთულის შემცირებულად წარმოჩენა; Playwright-ით მსხვილი პლატფორმების სკრეპინგისას მომსახურების პირობების (ToS) დარღვევის რისკების გამოტოვება; და აგენტური ამოცანებისთვის მოდელების შერჩევის არაზუსტი/ზოგადი ინსტრუქციები.

**პედაგოგიკური სტრუქტურა (პუნქტი 4):** ფუნდამენტური სტრუქტურული პრობლემა არის სესიების დროს პრაქტიკული ჩართულობის (hands-on) 0%-იანი მაჩვენებელი. დემონსტრაციებს მუდმივად ასრულებს ლექტორი, ხოლო მონაწილეები მხოლოდ აკვირდებიან. აქტიური პრაქტიკის გარეშე, ინფორმაციის ათვისების მაჩვენებელ
ბის დროს პრაქტიკული ჩართულობის (hands-on) 0%-იანი მაჩვენებელი. დემონსტრაციებს მუდმივად ასრულებს ლექტორი, ხოლო მონაწილეები მხოლოდ აკვირდებიან. აქტიური პრაქტიკის გარეშე, ინფორმაციის ათვისების მაჩვენებელი იქნება დაბალი, ხოლო თავდაჯერებულობა — ხელოვნურად მაღალი, ზუსტად იქამდე, სანამ მონაწილეები დამოუკიდებლად მუშაობას არ დაიწყებენ და პირველივე წინაღობას არ წააწყდებიან.

**ტემპი (პუნქტი 5):** საერთოდ არ არის გამოყოფილი დრო მასალის გაგების გადასამოწმებლად, მონაწილეთა პრაქტიკისთვის ან სტრუქტურირებული რეფლექსიისთვის. სესია ტოვებს „ინფორმაციულად სრული, მაგრამ სწავლების კუთხით დაუსრულებელი“ პროცესის შთაბეჭდილებას — ბევრი რამ არის ნაჩვენები, მაგრამ ცოტა რამ არის რეალურად ათვისებული.

**მომავალი სესიისთვის მომზადება (პუნქტი 6):** პრიორიტეტული ქმედებები: სესიის დამხმარე დოკუმენტის (companion document) შექმნა წინასწარ; ერთი სავალდებულო პრაქტიკული სავარჯიშოს დიზაინი; შეცდომების მართვის გზამკვლევის მომზადება; და დროის განაწილების ხილული სტრუქტურის დანერგვა.

### 8. ბაზრის რელევანტურობა ქართული კონტექსტისთვის

**მაღალი რელევანტურობა:**
კურსი პირდაპირ პასუხობს ქართული კომპანიებისთვის AI-ს დანერგვის ყველაზე დიდ ბარიერს: ნაპრალს იმის ცოდნას (რომ AI არსებობს) და იმის ცოდნას შორის (თუ როგორ დაინერგოს ის უსაფრთხოდ ორგანიზაციაში). კორპორატიული უსაფრთხოების განხილვა (მონაცემთა სუვერენიტეტი, სენსიტიური და არასენსიტიური სამუშაოს გამიჯვნა) ზუსტად ის არის, რაც ქართულ ფინანსურ ინსტიტუტებს, სახელმწიფო ორგანიზაციებსა და მსხვილ კორპორაციებს სჭირდებათ.

სასტუმროების მონიტორინგის აგენტის მაგალითი (ტარიელის პროექტი) ასახავს ქართული ტურიზმის სექტორის რეალურ გამოწვევას. "Vibe coding"-ის ფილოსოფია განსაკუთრებით კარგად ერგება ქართულ ბიზნეს-გარემოს, სადაც AI წიგნიერება იზრდება, მაგრამ ტექნიკური AI ტალანტების დეფიციტია.

**საშუალო რელევანტურობა:**
ჩინური AI მოდელები გლობალურად ხელმისაწვდომია, მაგრამ მათი ქართული ენის მხარდაჭერა დიდწილად გადაუმოწმებელია და სავარაუდოდ სუსტია. მათი, როგორც იაფი ალტერნატივის პოზიციონირება ლოგიკურია, მაგრამ მათი პრაქტიკული ღირებულება ქართულენოვანი ბიზნეს-აპლიკაციებისთვის არ არის ვალიდირებული და შესაბამისი გაფრთხილებებით უნდა მიეწოდოს აუდიტორიას.

**ქართული ბაზრის დაფარვის ხარვეზები:**
- **ქართულენოვანი AI-ს წარმადობის ანალიზი:** ეს ალბათ ყველაზე პრაქტიკული და მნიშვნელოვანი თემაა ქართული ბიზნესებისთვის. რომელი მოდელი აგენერირებს გამართულ, კულტურულად რელევანტურ ქართულს? რომელი უფრო მეტად ჰალუცინირ
ანი AI-ს წარმადობის ანალიზი:** ეს ალბათ ყველაზე პრაქტიკული და მნიშვნელოვანი თემაა ქართული ბიზნესებისთვის. რომელი მოდელი აგენერირებს გამართულ, კულტურულად რელევანტურ ქართულს? რომელი უფრო მეტად ჰალუცინირებს ქართულად? ეს მონაცემები კურსში არ ჩანს.
- **ქართული სარეგულაციო გარემო:** პერსონალურ მონაცემთა დაცვის შესახებ საქართველოს კანონი, სექტორული რეგულაციები ფინანსებისა და ჯანდაცვისთვის — ეს საერთოდ არ არის დაფარული. კორპორატიული მონაწილეებისთვის, რომლებიც მომხმარებელთა მონაცემებზე მომუშავე აგენტებს ქმნიან, ეს არ არის არჩევითი თემა.
- **ფასის მიმართ მგრძნობელობა:** ქართული მცირე და საშუალო ბიზნესი (SME) ფასების მიმართ მგრძნობიარე გარემოში ოპერირებს. თვეში $100-$200-იანი Claude-ის რეკომენდაცია მოცემულია უფასო ან იაფი ალტერნატივების შედარების გარეშე, რომლებსაც იგივე ამოცანების 80%-ის შესრულება შეუძლიათ.
- **ინტეგრაცია ქართულ ბიზნეს-ინფრასტრუქტურასთან:** ქართული ბანკების API-ები, გადახდის პროვაიდერები, მთავრობის e-სერვისები — აგენტები, რომლებიც უნიკალურ ქართულ სისტემებთან ინტეგრირდებიან, ყველაზე დიდ ლოკალურ ღირებულებას ქმნიან და ეს თემა გამოტოვებულია.

### 9. კონკურენტული ანალიზი — რას ასწავლიან სხვები და რა აკლია ამ კურსს

**1. Prompt Engineering, როგორც სისტემური დისციპლინა**
ამ კურსში იგულისხმება, რომ მონაწილეებს უკვე შეუძლიათ ეფექტური "პრომპტების" წერა. ეს დაშვება არასწორი და საშიშია. განსხვავება კარგად სტრუქტურირებულ system prompt-სა და ჩვეულებრივ ინსტრუქციას შორის არის განსხვავება საიმედო და არასტაბილურ აგენტს შორის. Chain-of-thought, few-shot მაგალითები, როლების მინიჭება, შეზღუდვების დაწესება — ეს მაღალი გავლენის მქონე უნარებია, რომელთა სწავლებაც აუცილებელია.

**2. RAG (Retrieval-Augmented Generation) არქიტექტურა**
ვექტორული ბაზები ნახსენებია, მაგრამ RAG, როგორც სრული არქიტექტურული პატერნი (დოკუმენტების ემბედინგი, ვექტორულ ბაზაში შენახვა, კონტექსტის ამოღება და პრომპტში ინექცია) არასდროს არის წარმოდგენილი ერთიან კონცეფციად. ეს არის ფუნდამენტური პატერნი კომპანიის შიდა ცოდნაზე დაფუძნებული აგენტების შესაქმნელად.

**3. AI პროექტის ROI-ს დათვლა და ბიზნეს-ქეისის განვითარება

### 13. ერთი ყველაზე მნიშვნელოვანი სარეკომენდაციო შეტყობინება ლექტორს

**თორნიკე, კურსს აქვს სტრუქტურული ხარვეზი, რომელიც შენი ნამდვილი ექსპერტიზის ღირებულებას აკნინებს: შენ გასცემ ინფორმაციას იქ, სადაც უნარების კურსს უნდა ასწავლიდე.**

არსებობს ფუნდამენტური სხვაობა ლექც
**თორნიკე, კურსს აქვს სტრუქტურული ხარვეზი, რომელიც შენი ნამდვილი ექსპერტიზის ღირებულებას აკნინებს: შენ გასცემ ინფორმაციას იქ, სადაც უნარების კურსს უნდა ასწავლიდე.**

არსებობს ფუნდამენტური სხვაობა ლექციასა (სადაც ლექტორი გადასცემს ინფორმაციას) და ტრენინგ-კურსს (სადაც მონაწილეები ივითარებენ უნარებს) შორის. შენი მსმენელები თითოეული სესიიდან მიდიან შესანიშნავი კონტენტის მოსმენის შემდეგ, მაგრამ ნულოვანი პრაქტიკით. 90-წუთიან სესიაზე მოსმენილი ინფორმაცია პრაქტიკის გარეშე 48 საათში ტოვებს მხოლოდ 10-20%-იან ათვისების მაჩვენებელს (ებინგჰაუსის დავიწყების მრუდი). მოსმენილი და პრაქტიკაში გამოცდილი — 60-70%-ს. ხოლო მოსმენილი, გამოცდილი და პირად პროექტზე მორგებული — 90%+-ს.

შენი ყველაზე ძლიერი მსმენელი შეძლებს Playwright-ის დემოს გამეორებას, რადგან ყურადღებით გიყურებდა. შენი საშუალო მსმენელი — ვერა; მან დაინახა როგორ მოხდა ეს, მაგრამ თავად არ გაუკეთებია. შენმა ნაკლებად თავდაჯერებულმა მსმენელმა კი შეიძლება საერთოდ დაკარგა რწმენა, რომ ამას შეძლებს, რადგან ნაპრალი „ლექტორი ამას ძალისხმევის გარეშე აკეთებს“ და „მე ეს თავად უნდა გავაკეთო“ შორის გადაულახავი ჩანს პირველი მართული პრაქტიკული მცდელობის გარეშე.

გამოსავალი სხვა კონტენტის სწავლება არ არის. შენი კონტენტის შერჩევა 2025 წლისთვის დიდწილად სწორია. გამოსავალი სტრუქტურულია: ყოველი სესია უნდა შეიცავდეს მინიმუმ 15 წუთიან მართულ პრაქტიკულ აქტივობას მონაწილეების მხრიდან — არა ყურებას, არა მოსმენას, არამედ კეთებას. თუ ეს ნიშნავს სესიაზე ნაკლები თემის გავლას, გაიარე ნაკლები თემა. ერთი პრაქტიკაში გამოცდილი თემა უდრის ათ უბრალოდ წარმოდგენილ თემას.

შენი ექსპერტიზა არ არის შემაფერხებელი ფაქტორი. შემაფერხებელია შენი სწავლების არქიტექტურა. გადააწყვე სესიები მონაწილეთა პრაქტიკული ქმედებების გარშემო და შენ შექმნი რაღაც მართლაც იშვიათს: კურსს, რომელიც აწარმოებს ქართველ ბიზნეს-მენეჯერებს, რომლებსაც დამოუკიდებლად შეუძლიათ AI აგენტების აწყობა და დანერგვა. ეს მიღწევადია — მაგრამ მხოლოდ იმ შემთხვევაში, თუ პრეზენტაციიდან პრაქტიკაზე სტრუქტურულ გადასვლას განახორციელებ.
